# ADR-001 — API Gateway as Single External Entry Point

**Date:** 2024-01-10  
**Status:** Accepted  
**Deciders:** Platform Architecture Team

---

## Context

The BankOS platform consists of multiple microservices. External clients (web app, mobile) need to reach these services. Without a central entry point, every service would need to:

- Validate JWT tokens independently (N calls to Keycloak's JWKS endpoint per request)
- Implement CORS policy independently (N places to misconfigure)
- Rate limit independently (per-instance, easily bypassed by targeting multiple instances)
- Expose its port externally (widening the attack surface)

## Decision

We deploy a **Spring Cloud Gateway** instance as the sole external entry point. All API traffic enters through port 8080 of the gateway. No other service is reachable from outside the cluster.

## Options Considered

**Option A: No gateway — direct service exposure**
- Each service gets its own Ingress rule
- Pros: simpler, fewer moving parts
- Cons: N×M duplication of auth/CORS/rate-limit, wide attack surface

**Option B: Nginx as a dumb reverse proxy**
- Pros: extremely lightweight
- Cons: no JWT validation, no circuit breaking, limited programmability

**Option C: Spring Cloud Gateway (chosen)**
- Pros: reactive, JWT validation built-in, circuit breaker integration, programmable filters
- Cons: one more JVM process, Redis dependency for distributed rate limiting

**Option D: Kong / AWS API Gateway (managed)**
- Pros: fully managed, enterprise support
- Cons: cost, cloud lock-in, less transparent for a self-hosted portfolio demo

## Consequences

**Positive:**
- Single point for JWT validation — downstream services trust the forwarded token
- Centralised rate limiting via Redis (works across replicas)
- One CORS config, one security policy surface
- Easy to add cross-cutting concerns (correlation ID, logging, request tracing)
- Network policies can restrict intra-cluster traffic to known paths

**Negative:**
- Single point of failure if not replicated (mitigated: minReplicas=2, HPA)
- Added latency (~5-10ms per request for JWT validation and routing)
- Redis dependency for rate limiting (mitigated: gateway degrades gracefully if Redis is down — rate limiting is disabled, not the entire gateway)
