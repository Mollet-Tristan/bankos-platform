# ADR-003 — Synchronous REST for Account Debit/Credit Operations

**Date:** 2024-01-12  
**Status:** Accepted  
**Deciders:** Platform Architecture Team

---

## Context

The Transaction Service needs to debit the source account (and credit the target for transfers) via the Account Service. Two communication patterns are possible: synchronous REST or asynchronous messaging (Kafka).

The same question arises for the API Gateway → downstream services communication.

## Decision

**Transaction Service → Account Service: synchronous REST (HTTP)**

The debit/credit calls are synchronous. The Transaction Service waits for a response before recording the transaction outcome.

## Decision Drivers

1. **The user expects an immediate answer.** When a user submits a payment, they need to know if it succeeded or failed before leaving the page. "Your payment will be processed eventually" is not acceptable for a debit operation.

2. **The Account Service enforces invariants that must be known before proceeding.** If the debit fails (InsufficientFunds, AccountFrozen), the Transaction Service must know *immediately* to mark the transaction as FAILED — not discover it later via an event.

3. **The Saga pattern handles failures synchronously.** The compensation flow (credit back the source after a failed downstream step) requires knowing the debit result before deciding whether to compensate.

## Why Not Async (Kafka)?

If we published a "DebitRequested" event to Kafka:
- The Transaction Service would need to wait for a "DebitConfirmed" or "DebitRejected" event — effectively a synchronous request-reply pattern implemented over async messaging, which adds complexity without benefit
- The user would receive a "payment submitted" response with no guarantee of outcome
- Implementing request-reply over Kafka requires correlation IDs, timeouts, and response topics — more complexity than REST

**The rule of thumb: use async when the caller does NOT need the result. Here, the caller absolutely needs the result.**

## Consequences

**Positive:**
- Simple, well-understood pattern
- Immediate feedback to the user
- Clear error propagation (HTTP 422 → InsufficientFundsException → FAILED transaction)
- Easy to implement circuit breaker with Resilience4j

**Negative:**
- Temporal coupling: if Account Service is down, Transaction Service cannot process payments
- Mitigated by: circuit breaker (fail fast after N failures), retry with exponential backoff for transient errors

**Scope boundary:** only debit/credit operations are synchronous. Post-transaction notifications (TransactionCompleted event → NotificationService) are async. See ADR-004.
