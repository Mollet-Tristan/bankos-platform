# ADR-004 — Asynchronous Kafka Events for Post-Transaction Notifications

**Date:** 2024-01-12  
**Status:** Accepted  
**Deciders:** Platform Architecture Team

---

## Context

After a transaction completes (or fails, or is compensated), several downstream consumers need to react:
- **NotificationService** — send email/SMS to the account owner
- **ReportingService** — update local analytics database
- (Future) **AuditService** — append to immutable audit log
- (Future) **FraudDetectionService** — analyse transaction patterns

The question is how the Transaction Service should communicate these outcomes to downstream consumers.

## Decision

**Transaction Service publishes domain events to Kafka. Consumers subscribe independently.**

Topics:
- `bankos.transaction.events` — TransactionCreated, TransactionCompleted, TransactionFailed, TransactionCompensated
- `bankos.account.events` — AccountCreated, AccountDebited, AccountCredited, AccountStatusChanged

## Decision Drivers

### Why Async Here (contrast with ADR-003)?

The critical difference vs the debit/credit operation:
- **The user does not need to wait for the notification to be sent** before the payment response is returned
- **The notification can be delayed** by seconds or minutes without harming the user experience
- **The notification consumer can be down** without impacting the payment flow

This is the fundamental rule: **use async when the caller does not need the result**.

### Kafka Specifically

**Option A: Direct REST call (Transaction → Notification)**
- Temporal coupling: if Notification Service is down, the debit still completes but notification fails silently
- The Transaction Service must know about the Notification Service — tight coupling
- Adding a new consumer (Audit, Fraud) requires changing the Transaction Service

**Option B: Kafka (chosen)**
- Zero coupling: Transaction Service publishes; it doesn't know who consumes
- Notification Service can be down for minutes — Kafka retains events, they process on recovery
- Adding a new consumer requires zero changes to the Transaction Service
- Natural audit trail: the topic is an append-only log of all financial events
- Replay capability: ReportingService can replay from the beginning to reconstruct history

**Option C: Message broker (RabbitMQ)**
- Better for task queues, not event streams
- No replay capability (messages are deleted after consumption)
- Less suited for "multiple consumers, each with independent offset"

## Topic Design

```
bankos.transaction.events
  Partition key: transactionId
  → guarantees ordering per transaction
  → all events for tx-123 go to the same partition

bankos.account.events  
  Partition key: accountId
  → ordering per account
```

## Consumer Groups

Each service has its own consumer group — they each receive all events independently:
- `notification-service` → processes TransactionCompleted, TransactionFailed
- `reporting-service` → processes all events for analytics
- (Future) `audit-service` → processes all events for immutable log

## Consequences

**Positive:**
- Transaction Service has no runtime dependency on Notification Service
- Notification failures do not affect payment throughput
- Kafka replay enables historical report reconstruction
- Adding consumers requires zero changes to producers
- Natural event log for debugging: `kafka-console-consumer --from-beginning`

**Negative:**
- Eventual consistency: notification arrives slightly after transaction commit
  (acceptable — no user expects instant email, milliseconds at most)
- Operational complexity: Kafka cluster must be managed
- At-least-once delivery: consumers must be idempotent (handled via sourceEventId check)

## Idempotency Note

Kafka guarantees at-least-once delivery. Consumers must handle duplicate messages:
- NotificationService checks `sourceEventId` before creating a notification
- ReportingService uses SQLite `INSERT OR IGNORE` on the transaction ID
- Both have DB-level unique constraints as final safety net
