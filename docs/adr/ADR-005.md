# ADR-005 — Rust for the Reporting Service

**Date:** 2024-01-15  
**Status:** Accepted  
**Deciders:** Platform Architecture Team  

---

## Context

The BankOS platform needs a reporting component to compute financial analytics: transaction volumes, failure rates, per-account activity, and period-over-period trends.

The question is: what stack should this component use?

All other services in the platform are Kotlin + Spring Boot. Continuing with that stack would be the path of least resistance. However, the reporting workload has a distinct profile that warrants re-evaluation.

### Workload characteristics

| Characteristic | Account/Transaction Service | Reporting Service |
|---|---|---|
| Workload type | I/O-bound (DB reads/writes) | CPU-bound (aggregation loops) |
| Latency profile | Synchronous, < 200ms SLA | Batch/CLI, minutes acceptable |
| Deployment model | Long-running HTTP server | CLI binary + background daemon |
| Runtime requirement | JVM | None (native binary) |
| Startup time | 1-3s (JVM warmup) | < 50ms (native) |
| Memory profile | 200-500MB (JVM heap) | < 20MB RSS |

### Options considered

**Option A: Kotlin + Spring Boot (same as other services)**

- Pros: same team knowledge, same toolchain, consistent codebase
- Cons: JVM startup penalty for CLI use, high memory for aggregation work, GC pauses during large aggregation loops

**Option B: Kotlin + GraalVM Native Image**

- Pros: native binary from Kotlin, same language
- Cons: GraalVM native compilation is complex to configure, limited reflection support, long build times (10+ minutes), less predictable behavior than pure native

**Option C: Go**

- Pros: simple language, fast compilation, good concurrency, small binaries
- Cons: no compile-time guarantees for memory safety, GC (though low-latency), less expressive type system for domain modeling

**Option D: Rust**

- Pros: no GC pauses, smallest binary size, excellent performance, WASM-compilable, strong type system for domain modeling, `rust_decimal` for exact financial arithmetic
- Cons: steeper learning curve, longer initial development time, different ecosystem from the rest of the platform

---

## Decision

**We choose Rust (Option D).**

### Primary reasons

**1. No GC pauses for aggregation loops**

When aggregating 1M transactions across a period, a JVM GC pause of 50-200ms at an unpredictable point in the loop degrades user experience. Rust's deterministic memory management means the loop runs from start to finish without interruption.

**2. CLI startup time**

A CLI tool that takes 2 seconds to start (JVM warmup) is frustrating for interactive use and slow in CI pipelines. `bankos-report` starts in under 50ms.

**3. Exact decimal arithmetic**

`rust_decimal::Decimal` provides exact decimal arithmetic without floating-point rounding. This is non-negotiable for financial calculations. The JVM equivalent (`java.math.BigDecimal`) is verbose; Rust's `rust_decimal` is ergonomic and has the `dec!()` macro for literals.

**4. WASM future path**

The pure domain functions in `src/domain/aggregation.rs` have no I/O, no async, and no OS dependencies. They can be compiled to WASM and run in a browser for client-side analytics. This path is not available without significant rework in a JVM context.

**5. Portfolio signal**

This is an explicit portfolio consideration. A Solution Architect who can evaluate tradeoffs and choose Rust for a specific workload — rather than defaulting to the same stack everywhere — demonstrates architectural maturity.

### Secondary reasons

- Binary size: 5MB stripped vs 80MB fat JAR
- Memory: < 20MB RSS vs 200-500MB JVM heap
- Portability: single binary with no runtime dependency (no JRE required)

---

## Consequences

### Positive

- CPU-bound aggregation with predictable performance
- Fast CLI startup for developer and CI use
- Small deployment artifact (binary only, no JVM)
- Potential WASM compilation for browser-side analytics
- Demonstrates polyglot capability in the portfolio

### Negative

- Different toolchain (Cargo vs Gradle) — CI pipeline must support both
- Team must learn Rust (steeper curve than Kotlin)
- Different error handling model (Result/? vs try/catch)
- `async_trait` crate needed for async trait methods (limitation being addressed in Rust)
- rdkafka requires librdkafka or cmake-build feature (adds to build time)

### Mitigations

- `CLAUDE.md` documents the Rust-specific rules for AI-assisted development
- `async_trait` will be replaceable with native async traits when stabilized (Rust 2024+)
- The `cmake-build` feature for rdkafka keeps the build self-contained (no system librdkafka)

---

## Scope boundary

Rust is used **only** for the reporting service. It is not a recommendation to migrate other services. The Account, Transaction, and Notification services remain Kotlin+Spring where the team expertise is highest and the workload fits perfectly.

The principle: **use the right tool for the workload**, not the same tool for everything.

---

## Review

This decision will be reviewed if:
- The Kotlin/Spring team cannot allocate time to maintain Rust components
- Performance requirements change such that JVM becomes sufficient
- GraalVM Native Image matures to the point where Option B becomes viable
