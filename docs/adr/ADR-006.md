# ADR-006 — Database per Service

**Date:** 2024-01-13  
**Status:** Accepted  
**Deciders:** Platform Architecture Team

---

## Context

The BankOS platform has four services that need persistence. The simplest approach would be a single shared PostgreSQL database. This is the most common pattern in monolithic architectures and even in early-stage microservices.

## Decision

**Each service owns its own database.** No service reads another service's tables directly.

| Service | Database |
|---|---|
| account-service | `bankos_accounts` (PostgreSQL) |
| transaction-service | `bankos_transactions` (PostgreSQL) |
| notification-service | `bankos_notifications` (PostgreSQL) |
| reporting-service | `bankos-reports.db` (SQLite, local) |

## Options Considered

**Option A: Shared database (one PostgreSQL instance, shared schema)**
- Pros: simple, single backup target, easy cross-table JOINs
- Cons: schema coupling (a migration in one service breaks others), shared connection pool (one service can starve others), cannot scale databases independently

**Option B: Shared database, separate schemas**
- Pros: isolation per service, single PostgreSQL instance
- Cons: still shared connection pool, `GRANT` management complexity, temptation to JOIN across schemas

**Option C: Database per Service (chosen)**
- Pros: true isolation — Account Service schema changes don't affect Transaction Service, independent scaling, can use different DB technologies per service (SQLite for reporting), independent backups, independent tuning
- Cons: no cross-service JOINs, data consistency across services requires application-level coordination (sagas, events)

## Consequences

### Positive

**Independent deployability:** Account Service can migrate its schema, scale its database, or change its ORM without coordinating with Transaction Service.

**Technology fit per workload:**
- Account Service → PostgreSQL with optimistic locking (OLTP, strong consistency)
- Reporting Service → SQLite (CLI, no server, single file)
- (Future) AuditService → append-only store (EventStore, Cassandra)

**Independent scaling:** Account Service database can be given more connections/memory without affecting others.

**Failure isolation:** If Transaction DB is overloaded, Account DB is unaffected.

### Negative

**No JOINs across services.** If a query needs account data AND transaction data, one service must call the other via REST or both must maintain a local read model. This is a conscious trade-off.

**Data consistency requires application-level coordination.** A transfer that debits account A and credits account B cannot use a single DB transaction. This is solved by the Saga pattern (ADR-007).

**More infrastructure.** Three PostgreSQL instances instead of one. Mitigated by using a managed PostgreSQL service (RDS, Cloud SQL) per environment, which reduces operational burden.

### The practical boundary

Services NEVER share a database. If Service A needs data owned by Service B:
1. **API call** — Service A calls Service B's REST endpoint (synchronous, small data)
2. **Event subscription** — Service A subscribes to Service B's Kafka topic and maintains a local read model (asynchronous, for queries that would be too slow via REST)
3. **Read model / projection** — ReportingService reads Transaction events from Kafka and stores a denormalised view in SQLite
