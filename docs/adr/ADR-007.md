# ADR-007 — Saga Pattern for Distributed Transaction Management

**Date:** 2024-01-14  
**Status:** Accepted  
**Deciders:** Platform Architecture Team

---

## Context

A bank transfer touches two services: Account Service (debit source, credit target) and Transaction Service (record the movement). These cannot be wrapped in a single ACID transaction — they use separate databases (ADR-006) on separate processes.

Classic distributed transaction protocols (2PC/XA) provide ACID guarantees across services but with severe trade-offs:

| Problem | 2PC/XA Impact |
|---|---|
| Coordinator failure | All participants blocked indefinitely |
| Network partition | Locks held on all participants |
| Availability | Reduced — all participants must be up to commit |
| Scalability | Poor — coordinator is a bottleneck |

## Decision

**We use the Choreography Saga pattern with explicit compensating transactions.**

There is no central saga coordinator. The Transaction Service orchestrates the flow locally:
1. Persist transaction (PENDING)
2. Call Account Service for debit (REST, synchronous)
3. On success: mark COMPLETED, publish TransactionCompleted event
4. On business failure (InsufficientFunds): mark FAILED, no compensation needed (debit never happened)
5. On infrastructure failure after debit: mark COMPENSATED, issue compensating credit to Account Service

## Saga State Machine

```
PENDING
  │
  ▼ startProcessing()
PROCESSING ──── debit OK ──────────────────── COMPLETED
  │                                               │
  ├──── business rejection (422) ──────────── FAILED
  │     (no compensation — debit rejected)
  │
  └──── infrastructure failure (5xx/timeout) ── COMPENSATED
        → compensating credit issued
```

## Why Choreography vs Orchestration?

**Orchestrated Saga:** a dedicated Saga Orchestrator service drives each step, explicitly calling services in sequence.
- Pros: clear overview of the saga in one place, easier to add steps
- Cons: new service to build and operate, single point of failure, network hop for each saga step

**Choreography Saga (chosen):** each service reacts to events and executes its local step.
- Pros: no central coordinator, services are loosely coupled
- Cons: the saga logic is distributed across services (harder to visualise the full flow)

For this platform's scope (2-3 services per saga), choreography is simpler and avoids the operational overhead of an orchestrator.

## Compensation Design

**When is compensation triggered?**
Only when a debit succeeds but a subsequent step fails (e.g., credit target fails, DB write fails). A business failure (InsufficientFunds → 422) means the debit was rejected — no compensation needed.

**What does compensation do?**
Issues a credit call to the Account Service for the same amount as the original debit, with reference `{transactionId}-compensation`.

**What if compensation itself fails?**
This is the hardest case. The system:
1. Logs `CRITICAL` — this requires human intervention
2. Publishes to a dead-letter Kafka topic (TODO: ADR-008)
3. Sets status to COMPENSATED — the record exists for audit

A truly robust solution would use the Outbox pattern (TODO: ADR-009) to guarantee that either the debit happens AND the compensation event is stored, or neither.

## Idempotency

The Saga state machine doubles as an idempotency guard:
- `startProcessing()` only succeeds from PENDING → rejects concurrent retries
- The DB unique constraint on `idempotency_key` rejects concurrent inserts
- Each saga step uses a stable reference (`{transactionId}-debit`, `{transactionId}-credit`) so Account Service can safely handle retries

## Known Limitations (Documented Honestly)

1. **Double-credit risk:** if compensation fails AND the system crashes before logging, the source account is debited with no record. Mitigated by: CRITICAL log, dead-letter topic (TODO)

2. **No Outbox pattern:** if the JVM crashes between DB write and Kafka publish, the event is lost. The database record exists (COMPLETED) but no notification is sent. The Outbox pattern would fix this.

3. **No saga timeout:** a PROCESSING transaction that never completes due to a deadlock would stay PROCESSING forever. A background sweeper job (TODO) should detect and resolve stuck sagas.

These limitations are documented rather than hidden — a production system would address them progressively.
