# ADR-016 — Deployment Strategy: Kubernetes + Kustomize (dev) + Helm (prod)

**Date:** 2024-01-20  
**Status:** Accepted  
**Deciders:** Platform Architecture Team

---

## Context

The BankOS platform needs a deployment strategy that works across three environments: local development (laptop), staging (shared cluster), and production (managed K8s). The strategy should balance:
- Developer experience (fast feedback loop locally)
- Ops safety (no accidental production deployments)
- DRY configuration (changes to resource limits should propagate automatically)
- Auditability (every deployment is traceable)

## Decision

**Dual strategy:**
- **Kustomize** for local and dev environments (base + overlays)
- **Helm** for staging and production (parameterised, versioned releases)

Both use the same base Kubernetes manifests. Kustomize patches them for local; Helm templates generate them for production.

## Why Not One Tool for Everything?

### Kustomize only

- Great for overlay-based environment customisation
- Poor for parameterised releases (no `--set image.tag=1.2.3` equivalent)
- Helm's `helm upgrade --atomic --rollback` is better for production safety

### Helm only

- Great for production parameterisation
- `helm install` is heavyweight for local dev
- YAML templating syntax (`{{- if .Values.foo }}`) is verbose and error-prone for simple cases

### Both (chosen)

Each tool is used where it is strongest:
- Kustomize: simple overlay for local (`replicas: 1`, local image tags) — readable YAML patches
- Helm: production releases with `--atomic` (automatic rollback on failure), release history, `helm diff` for change review

## Kustomize Design

```
infra/kubernetes/
  base/               ← production-like defaults
  overlays/
    local/            ← 1 replica, local image tags, no HPA
    staging/          ← 2 replicas, staging config
    production/       ← 3 replicas, strict resource limits
```

Apply: `kubectl apply -k infra/kubernetes/overlays/local`

## Helm Design

```
infra/helm/bankos-platform/
  Chart.yaml
  values.yaml         ← defaults
  templates/
    deployment.yaml   ← single DRY template for all 4 services
```

A single Deployment template with `{{- define "bankos.deployment" }}` renders all four services. This demonstrates Helm's DRY templating value vs raw YAML.

## Key Principles

**Zero-downtime rollouts:**
- `RollingUpdate` with `maxUnavailable: 0` — never take pods offline before new ones are ready
- `PodDisruptionBudget` — protect against voluntary disruptions (node drain)

**Resource governance:**
- `ResourceQuota` at namespace level — prevents runaway consumption
- `LimitRange` — default limits for pods that don't specify them
- `requests` and `limits` on every container — no unbounded pods

**Security:**
- `runAsNonRoot: true` on all pods
- `NetworkPolicy` restricting intra-cluster communication to known paths
- Secrets never in Git — bootstrapped via `bootstrap-secrets.sh` from a secrets manager

**Observability:**
- `prometheus.io/*` annotations on all pods — auto-discovery without manual scrape config
- Liveness + readiness + startup probes — Kubernetes knows when a pod is actually ready

## Consequences

**Positive:**
- Developer can spin up the platform locally with one command
- Production deployments are atomic (Helm `--atomic` rolls back on failure)
- Deployment history is visible via `helm history bankos-platform`
- `helm diff` previews what will change before applying

**Negative:**
- Two tools to understand (Kustomize + Helm)
- Kustomize and Helm manifests can drift if not maintained together
- Mitigated: both tools use the same base manifests; the `deploy.sh` script enforces consistent usage
