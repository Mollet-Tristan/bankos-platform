# ADR-017 ‚Äî Observability: Prometheus + Grafana

**Date:** 2024-01-21  
**Status:** Accepted  
**Deciders:** Platform Architecture Team

---

## Context

Operating a distributed system without observability is flying blind. We need to answer:
- Is the platform healthy right now?
- Which service is causing latency?
- Did the Kafka consumer fall behind after the deployment?
- Did the circuit breaker open overnight?

## Decision

**Prometheus for metrics collection, Grafana for visualisation and alerting.**

All Spring Boot services expose `/actuator/prometheus` via Micrometer. Kubernetes pod annotations (`prometheus.io/scrape: "true"`) enable automatic discovery ‚Äî no manual scrape config per service.

## Options Considered

| Solution | Pros | Cons |
|---|---|---|
| **Prometheus + Grafana** (chosen) | Open source, rich ecosystem, self-hosted | Operational overhead |
| Datadog | Fully managed, excellent UX | Cost (per-host), financial data leaves on-premise |
| AWS CloudWatch | Native for AWS deployments | AWS lock-in, limited Kafka/JVM metrics |
| New Relic | Good APM, distributed tracing | Cost, data sovereignty |
| Elastic APM | Rich features, log integration | Complex to operate at scale |

**Why self-hosted for financial data:** metrics may contain account IDs, transaction amounts, and user activity patterns. Sending this to a SaaS metrics platform raises data sovereignty concerns. Self-hosted keeps financial telemetry on-premise.

## Three Pillars of Observability

### 1. Metrics (Prometheus + Micrometer)

Spring Boot Micrometer exposes:
- **HTTP metrics:** `http_server_requests_seconds_{count,sum,max,bucket}` ‚Äî latency histogram per endpoint
- **JVM metrics:** GC pauses, heap usage, thread count
- **HikariCP:** connection pool pending, active, idle counts
- **Resilience4j:** circuit breaker state, call counts by outcome
- **Kafka consumer:** lag per group/topic (via kafka-exporter)

Custom business metrics (added per service):
- `bankos_transaction_completed_total{type}` ‚Äî completed transactions by type
- `bankos_transaction_failed_total` ‚Äî failed transactions
- `bankos_transaction_compensated_total` ‚Äî saga compensations (critical to monitor)
- `bankos_notification_delivered_total{channel}` ‚Äî notification delivery
- `bankos_notification_permanently_failed_total` ‚Äî permanently failed notifications

### 2. Alerting (Prometheus AlertManager)

Alert design principles:
- **Alert on symptoms, not causes** ‚Äî `TransactionFailureRateHigh` not `AccountServiceDown`
- **Every alert is actionable** ‚Äî if ops cannot do anything, it is not an alert, it is noise
- **SLO-based** ‚Äî alerts fire before the SLO is breached, giving time to react

Key alerts defined in `monitoring/alerting/alerts.yaml`:
- `HighTransactionFailureRate` (>5% over 5min ‚Üí critical)
- `TransactionHighLatency` (P99 > 500ms ‚Üí warning)
- `TransactionCompensationSpike` (>0.1/s ‚Üí critical)
- `CircuitBreakerOpen` (any circuit open ‚Üí critical)
- `KafkaConsumerLag` (>1000 messages ‚Üí warning)
- `PodCrashLooping` ‚Üí critical
- `DatabaseConnectionPoolExhausted` ‚Üí critical

### 3. Dashboards (Grafana)

Two dashboards:
- **BankOS Overview** (`bankos-overview.json`) ‚Äî transaction rate, failure rate, P99 latency, circuit breakers, Kafka lag, JVM memory
- **Saga Health** (`saga-dashboard.json`) ‚Äî transaction outcome distribution, compensation rate, circuit breaker call breakdown, idempotent hit rate

### Logging (not in scope for this ADR)

Structured JSON logs (via Logback) with correlation IDs allow log aggregation. In a full setup, ELK (Elasticsearch + Logstash + Kibana) or Loki + Grafana would aggregate logs. Covered as future work.

### Distributed Tracing (not in scope for this ADR)

The gateway injects X-Correlation-Id, which propagates through all services. This is a minimum viable tracing foundation. Adding OpenTelemetry + Jaeger would provide full distributed traces. Documented as future work.

## Consequences

**Positive:**
- Zero SaaS cost ‚Äî runs entirely within the cluster
- Financial telemetry stays on-premise
- Rich ecosystem: Kafka exporter, PostgreSQL exporter, Redis exporter available
- Pull-based scraping ‚Äî services don't need outbound connectivity to a SaaS endpoint
- Grafana dashboards as code (JSON in Git) ‚Äî dashboards are versioned and reviewable

**Negative:**
- Operational overhead: Prometheus, AlertManager, Grafana must be kept running
- No built-in long-term storage ‚Äî Prometheus retention is typically 15 days. Thanos or Cortex needed for longer retention.
- No distributed tracing out of the box (see above)

## Sizing Note

For a portfolio demo, Prometheus runs with the default 15-day retention and 2GB storage. Production would add:
- Thanos for long-term storage (object storage: S3, GCS)
- Prometheus HA (two replicas with deduplication via Thanos)
- AlertManager HA (clustered)
